## はじめに

Flutterで「ビルド毎にいい感じにコードが調整されたらなぁ」と思うことはないでしょうか。本記事では、`const`を利用したコードの最適化について、実験的な見解を含む内容をまとめます。すべて筆者が調べた内容です。もし間違った情報や、より良い方法があれば、お知らせください。

---

Dartの `const` は**コンパイル時に値が確定する**ことを意味します。普段の開発では、Widgetのコンストラクタに付けるとか、lintルールで求められるとかいった印象かもしれません。

https://dart.dev/language/variables#final-and-const

コンパイル時に値が確定するということは、AOTコンパイラがコードを評価する時点で値が決まっているということです。ということは `const` を駆使することで、コンパイラに「このコードは到達不可能です」と伝えることもできるはずです。到達不可能であることが示されれば、tree-shakingの恩恵を受けることができるはずです。

本記事では、この「はず」を出発点に、`const` にできない値をどうやって回避し、tree-shakingまで橋をかけるかを考えます。

## `defaultTargetPlatform` の先例

`const`で宣言すれば、コードが到達不可能であることは一目瞭然です。

```dart
const bool isShown = false;

class SampleWidget extends StatelessWidget {
  const SampleWidget({super.key});

  @override
  Widget build(BuildContext context) {
    if (!isShown) {
      return const SizedBox.shrink();
    }

    return const NiceWidget();
  }
}
```

このコードをコンパイルすれば、早期リターンで `SizedBox` が返却されます。このため `NiceWidget` は、他のクラスから参照されていなければ、tree-shakingの対象になります。

---

では、`const` で宣言できない値の場合はどうでしょうか？ 参考になるのがFlutterの `defaultTargetPlatform` です。`defaultTargetPlatform` は、リリースビルドにおいてtree-shakingが効くことが明記されています。

https://github.com/flutter/website/issues/11666

実装を確認すると、次のように `@pragma` が付与されたgetterであることがわかります。

```dart
@pragma('vm:platform-const-if', !kDebugMode)
TargetPlatform get defaultTargetPlatform => platform.defaultTargetPlatform;
```

`@pragma` は、Dartのコンパイラに対して特定の最適化や挙動を促すためのアノテーションです。FirebaseMessagingの `@pragma('vm:entry-point')` は、Backgroundでメッセージを受信したケースのために書いたことがある方も多いのではないでしょうか。実は、たくさんあります。

https://github.com/dart-lang/sdk/blob/3.10.0/runtime/docs/pragmas.md

https://github.com/dart-lang/sdk/blob/3.10.0/pkg/compiler/doc/pragmas.md

このうち、`vm:platform-const-if` は、Flutterのプラットフォーム判定に特化したpragmaで、getterの戻り値をビルドターゲットのOSに基づいて、`const` として評価するようコンパイラに指示します。これにより、`defaultTargetPlatform` を条件にした分岐は、ビルドターゲットに応じて到達不可能なコードが除去されるようになります。要はターゲットではないOSのコードパスが、ビルド時に完全に消えるわけです。

https://github.com/dart-lang/sdk/issues/31969

---

ただ、多くのアプリエンジニアがやりたいことは、プラットフォームによる分岐ではなく**任意のユーザー定義**による分岐です。同じことを実現するには、どうすれば良いのでしょうか？

## `--dart-define`: コンパイル時定数をビルド時に遅延評価する

`--dart-define` は、Dartのコンパイル環境に対してコンパイル時定数を注入する仕組みです。FlutterKaigi 2025に参加された方は、色々なところで解説を聞いたと思います。^[気になる方はぜひFlutterKaigi 2025のセッション動画をチェックしてみてください。]

https://dart.dev/libraries/core/environment-declarations

値は `const String.fromEnvironment`、`const bool.fromEnvironment`、`const int.fromEnvironment` で受け取ります。これらは `const` コンストラクタであり、返される値はコンパイル時定数です。うれしいですね。

```shell
# 単一の値を渡す
flutter build apk --dart-define=BUILD_MODE=production

# ファイルから一括で渡す
flutter build apk --dart-define-from-file=config/production.env
```

公式ドキュメントでは次のように説明されています。

> Dart compilers can eliminate the code made unreachable due to control flow using the environment declaration values.

つまり `--dart-define` で渡した値は `const` 。それを利用した `if` 分岐が `false` に確定すれば、コンパイラはその分岐と本体を完全に除去できます。コンパイル時定数を、**ビルド時に遅延させて確定させられる**わけです。

これは強力です。dev/stg/prodの環境切り替えで、不要なコードをリリースビルドから除外できることを意味します。ただ、迂闊に実装すると、`const` の伝播が途切れてしまいます。

### `const bool` による分岐: 確実に効く

最もシンプルなパターンから。明らかに効果のあるパターンは、`const bool` を直接 `if` 文の条件にすることです。

```dart
const bool kEnableDebug = bool.fromEnvironment('ENABLE_DEBUG');
```

`kEnableDebug` はコンパイル時定数であり、`--dart-define=ENABLE_DEBUG=true` を渡さない限り `false` に確定します。^[`bool.fromEnvironment` のデフォルト値は `false` です。]

```dart
if (kEnableDebug) {
  // ENABLE_DEBUG=trueの場合のみ到達可能
  // falseの場合、この分岐全体がtree-shakingの対象になる
  Navigator.of(context).push(
    MaterialPageRoute(builder: (_) => const DebugPage()),
  );
}
```

この分岐内でのみ参照される `DebugPage` クラスやその依存先は、コンパイル時に参照されないことが明らかです。参照グラフから外れれば、tree-shakingの対象になります。このパターンは言語仕様上の `const` 式であるため、コンパイラの挙動に依存せず**確実に**動作します。ここまでは順調です。

### constが途切れる瞬間

実務では、dev/stg/prodのように3つ以上の環境があり、環境ごとに複数のフラグを派生させたいケースが一般的です。`const bool` を個別に `--dart-define` で渡すと数が膨れますし、また環境ごとに複数のフラグを生成したくなります。

自然な発想として、1つの `const String` からenumに変換し、そこから各フラグを派生させてみましょう。

```dart
const String _buildModeStr = String.fromEnvironment('BUILD_MODE');

enum BuildMode {
  development,
  staging,
  production,
}

BuildMode get buildMode => switch (_buildModeStr) {
  'development' => .development,
  'staging' => .staging,
  'production' => .production,
  _ => throw UnsupportedError('Unknown BUILD_MODE: "$_buildModeStr"'),
};

bool get kEnableDebugMenu => buildMode == .development;
```

一見うまくいきそうですが、**constの伝播が途切れています**。

`_buildModeStr` はコンパイル時定数です。しかし、getterを経由した時点で、呼び出し元からは戻り値が `const` であることが見えなくなります。

Dartでは関数の戻り値を `const` として宣言する仕組みが言語に存在しません。もし `const BuildMode get buildMode => ...` のような宣言ができれば、この問題は起きませんが、現状では不可能です。`vm:try-evaluate-as-constant` というpragmaが提案されていますが、まだopenのままです。

https://github.com/dart-lang/sdk/issues/50473

---

なお、enumを使わずに文字列を直接比較すれば、`const` の伝播は途切れません。

```dart
const bool isDev = _buildModeStr == 'development';
```

しかし、これだと環境ごとに複数のフラグを派生させるのが面倒です。`buildMode` のようなgetterを用意して、そこからフラグを派生させる方が、コードの見通しも良く、保守性も高いでしょう。この「保守性が高い実装にしたい」という要望 (たとえばswitch式にしたいという要望) が、`const` の伝播と相性が悪いのです。堅実に対応するのであれば、`const bool` を個別に用意するか、ロジックの中で文字列を直接比較するかのどちらかになります。

## `@pragma('vm:prefer-inline')` で橋を架ける

ここで先ほどの `defaultTargetPlatform` を思い返してみます。あちらは `vm:platform-const-if` でconst性を伝えていました。しかしこのpragmaはコンパイラがビルドターゲットのOSを知っていることが前提であり、`--dart-define` で渡すユーザー定義値には使えません。

ここで、筆者が注目したのが `@pragma('vm:prefer-inline')` です。inline展開を促すこのpragmaを、getterに付与してみましょう。

```dart
@pragma('vm:prefer-inline')
BuildMode get buildMode => switch (_buildModeStr) {
  'development' => .development,
  'staging' => .staging,
  'production' => .production,
  _ => throw UnsupportedError('Unknown BUILD_MODE: "$_buildModeStr"'),
};

@pragma('vm:prefer-inline')
bool get kEnableDebugMenu => buildMode == .development;
```

`prefer-inline` は、コンパイラに対してこの関数をインライン展開するよう促します。インライン展開とは、関数呼び出しを関数の中身で置き換えることです。何が起きるか、呼び出し元の視点で見てみます。

インライン展開**前**、コンパイラから見た `build` メソッド内のコードは次のようになっています。

```dart
if (kEnableDebugMenu) {
  Navigator.of(context).push(
    MaterialPageRoute(builder: (_) => const DebugPage()),
  );
}
```

`kEnableDebugMenu` がインライン展開されると、次のようになります。

```dart
if ((switch (_buildModeStr) {
  'development' => .development,
  'staging' => .staging,
  'production' => .production,
  _ => throw UnsupportedError('Unknown BUILD_MODE: "$_buildModeStr"'),
}) == .development) {
  Navigator.of(context).push(
    MaterialPageRoute(builder: (_) => const DebugPage()),
  );
}
```

`_buildModeStr` は `const String.fromEnvironment('BUILD_MODE')` なので、ビルド時に値が確定しています。たとえば `BUILD_MODE=production` なら、コンパイラはこの式を次のように評価します。

```dart
// "production" は "development" ではないので false
if (false) {
  // 到達不能コード
}
```

通常、inline展開は処理速度の向上を目的に行われますが、ここでは**constを届けるため**に利用しています。コンパイラが持っている最適化（定数畳み込み、到達不能コード除去）が効く状態に持ち込むための橋渡しです。

---

なお、この手法はDartチームが公式に保証したパターンではありません。コンパイラの最適化パスの組み合わせに依存しており、言語仕様上の保証ではありません。とはいえ、実験的に検証した限りでは、VM AOTコンパイラもdart2jsも、`prefer-inline` を付与したgetterをインライン展開し、その後の定数畳み込みで `const String` の等値比較を解決してくれました。**このパターンは、現行のコンパイラ実装では機能する**と言えるでしょう。

### チェーン全体にpragmaを付ける

記事を書くにあたり、改めてpragmaの有無でtree-shakingの結果がどう変わるかを検証しました。

結果として、単純な等値比較を返すだけのgetter（`bool get isDevelopment => buildMode == BuildMode.development;`）は、pragmaがなくてもコンパイラが自動的にインライン化しました。

一方、**switch式を含むgetter（`BuildMode get buildMode => switch (...) { ... };`）は、pragmaがなければインライン化されず、tree-shakingが効きませんでした**。switch式のgetterにpragmaを付けた場合のみ、development専用のクラスがビルドから除去されることを確認しています。

コンパイラにとって、単純な等値比較を返すgetterは十分に小さいため自動的にインライン化しますが、switch式を含むgetterは関数本体が大きくなるため、ヒントがなければインライン化しないようです。実際にpragmaが必須なのはswitch式を含むgetterですが、コンパイラの自動判断に依存するよりも、チェーン全体にpragmaを付けておく方が安全です。

---

Flutter Webもサポートする場合、dart2js用のpragmaも併記する必要があります。VM用（`vm:prefer-inline`）とdart2js用（`dart2js:prefer-inline`）は別の名前空間に属しており、片方だけではもう一方のコンパイラに認識されません。注意しましょう。^[Flutter Web開発者が一人でも増えることを願っています。]

```dart
const String _buildModeStr = String.fromEnvironment('BUILD_MODE');

enum BuildMode {
  development,
  staging,
  production,
}

@pragma('vm:prefer-inline')
@pragma('dart2js:prefer-inline')
BuildMode get buildMode => switch (_buildModeStr) {
  'development' => .development,
  'staging' => .staging,
  'production' => .production,
  _ => throw UnsupportedError('Unknown BUILD_MODE: "$_buildModeStr"'),
};

@pragma('vm:prefer-inline')
@pragma('dart2js:prefer-inline')
bool get kEnableDebugMenu => buildMode == .development;

@pragma('vm:prefer-inline')
@pragma('dart2js:prefer-inline')
bool get kEnableStagingTools => buildMode != .production;

@pragma('vm:prefer-inline')
@pragma('dart2js:prefer-inline')
bool get kEnableAnalytics => buildMode == .production;

@pragma('vm:prefer-inline')
@pragma('dart2js:prefer-inline')
String get kApiBaseUrl => switch (buildMode) {
  .development => 'https://dev.example.com',
  .staging => 'https://stg.example.com',
  .production => 'https://api.example.com',
};
```

### 確実性について

以下、私の学術的なバックグラウンドがこの辺りではないため、AIに解説を補助してもらいながらの検討です。誤りがあればご指摘ください。

---

`const bool` による直接分岐は言語レベルの `const` 式であるため、確実にtree-shakingが効きます。一方、`prefer-inline` によるアプローチには、不確実な点が二つあります。

1. インライン化されるか
2. インライン化後に定数畳み込みが効くか

先ほど述べたように、実用上は機能が期待できます。なぜ動作するのか、その根拠を2点述べます。

**1. `fromEnvironment` の解決はCFE（共通フロントエンド）で行われる**

`const String.fromEnvironment('BUILD_MODE')` は、VM AOTやdart2jsといったバックエンドに届く前に、CFEの段階で具体的な文字列定数に解決されます。CFEが `const String.fromEnvironment('BUILD_MODE')` を、たとえば `"production"` という具体的な文字列定数に解決してから、バックエンドがインライン化や定数畳み込みを行う、という流れです。

CFEはDartの全コンパイラが共有する共通層です。つまり、定数の解決はバックエンドの実装に依存しません。VMであってもdart2jsであっても、おそらくdart2wasmであっても、CFEが定数を解決してからバックエンドが最適化を行うという流れは共通しています。

**2. 両方のバックエンドが同じパイプライン構造を持つ**

VM AOTコンパイラもdart2jsも、「インライン化→定数伝播→デッドコード除去」の順で最適化を行っており、定数伝播の段階で文字列の等値比較（`"production" == "development"` → `false`）を解決する能力を持っています。

また、この順序が将来壊れるリスクは低いと考えられます。

「インライン化した後に定数伝播を走らせる」のはコンパイラ最適化の基本的なパイプラインです。これを崩すことはコンパイラ全体の最適化性能を悪化させる方向の変更であり、意図的に行う動機がありません。`prefer-inline` + 定数伝播という組み合わせは、このパターンのために特別に作られたものではなく、コンパイラが本来持っている汎用的な仕組みの上に乗っています。

---

とはいえ言語仕様上の保証ではない以上、**`--analyze-size` で検証すること**を推奨します。

https://github.com/koji-1009/zenn_dart_const_tree_shaking

サンプルプロジェクトを用意しておきました。ぜひpragmaの有無でコードが削除されるかどうかを確認してみてください。

```shell
flutter build apk \
  --release \
  --dart-define=BUILD_MODE=production \
  --analyze-size \
  --target-platform android-arm64
```

ビルドが完了すると、`*-code-size-analysis_*.json` が生成されます。このJSONファイルをVSCodeなどで開き、tree-shakingの対象としたクラス名（例: `DebugPage`）で検索してください。pragmaありのビルドでクラスが見つからなければ、tree-shakingが効いています。pragmaなしのビルドと比較すれば、pragmaの効果を直接確認できます。

## 将来の展望

`defaultTargetPlatform` のtree-shakingを実現した `vm:platform-const-if` は次のIssueで議論され、実装されました。議論の焦点はあくまで「プラットフォーム固有のtree-shaking」であり、`--dart-define` 由来の値をどう扱うかは議題に上がっていません。

https://github.com/dart-lang/sdk/issues/31969

ざっと読んでいただければという感じなのですが、議論は最初から最後まで**プラットフォーム固有のtree-shaking**に絞られています。`--dart-define` によるユーザー定義値の伝播は議題に上がっていません。

---

現時点で `--dart-define` の値をgetterで返しつつtree-shakingを期待する場合、`vm:prefer-inline` によるインライン展開が実質的に唯一のアプローチです。改めてになりますが、これは公式に設計されたパターンではありません。とはいえ `const bool` で個別にフラグを渡す以外の方法がない以上、実務上の選択肢として検討に値するでしょう。

根本的には、Dartに `const get` があれば、pragmaに頼る必要はなくなります。現時点では `vm:try-evaluate-as-constant` が最も私が求めるものに近く、また明らかに言語機能として解決されるのが最も望ましいです。不安な方は、Issueの行方を見守るのが良いでしょう。
